# MetricGuard — TiDB Cloud Database Setup Guide

## Overview

[TiDB Cloud](https://tidbcloud.com) is a fully managed, MySQL-compatible database service. MetricGuard uses a TiDB Cloud cluster to store all collected system metrics and application logs. This guide walks you through setting up a free-tier TiDB Serverless cluster for MetricGuard.

> TiDB Cloud exposes a **MySQL-compatible** wire protocol. No MongoDB drivers or connection strings are required — any MySQL client or SQLAlchemy MySQL dialect connects directly.

---

## Step 1: Create a TiDB Cloud Account

1. Go to [https://tidbcloud.com](https://tidbcloud.com)
2. Click **"Get Started for Free"**
3. Sign up with email or Google/GitHub account
4. Verify your email address

---

## Step 2: Create a Free Serverless Cluster

1. In the TiDB Cloud Console, click **"Create Cluster"**
2. Choose **"Serverless"** (free tier)
3. Select a **Cloud Provider** (AWS recommended)
4. Select a **Region** closest to your deployment:
   - For India: `Singapore (ap-southeast-1)`
   - For US: `Oregon (us-west-2)` or `Virginia (us-east-1)`
5. Name your cluster: `MetricGuardCluster`
6. Click **"Create"**

> ⏳ Cluster provisioning takes 30–60 seconds.

---

## Step 3: Create a Database User

1. In the cluster detail page, go to **"Connect"**
2. Click **"Create Password"** to auto-generate a secure password, **or** set one manually
3. Note the generated credentials:

| Field | Value |
|-------|-------|
| Username | `<auto-generated, e.g. 3BnFRkK7pMxxxxxx.root>` |
| Password | `your-secure-password-here` |
| Host | `gateway01.ap-southeast-1.prod.aws.tidbcloud.com` |
| Port | `4000` |
| Database | `metricguard_db` (create this in Step 4) |

> ⚠️ **Important**: Save the password immediately — TiDB Cloud shows it only once after generation.

---

## Step 4: Create the MetricGuard Database

Connect via any MySQL client and create the database:

```sql
CREATE DATABASE IF NOT EXISTS metricguard_db;
USE metricguard_db;
```

MetricGuard's SQLAlchemy models will automatically create all required tables (`metrics`, `anomalies`, `logs`) on first startup via:

```python
Base.metadata.create_all(bind=engine)
```

No manual table creation is needed.

---

## Step 5: Configure Network Access

TiDB Serverless clusters use TLS and **do not require IP whitelisting** for the Serverless tier — connections are authenticated by username/password over an encrypted tunnel.

For **Dedicated** clusters:

1. Go to **"Security"** → **"IP Access List"** in the cluster settings
2. Click **"Add IP Address"**
3. For development: add `0.0.0.0/0`
4. For production: add only your server's IP addresses

> ⚠️ **Security Note**: Restrict IP access to known addresses in production environments.

---

## Step 6: Get Your Connection Details

In the cluster detail page, click **"Connect"** → **"General"** tab. Your connection parameters will look like:

```
Host:     gateway01.ap-southeast-1.prod.aws.tidbcloud.com
Port:     4000
User:     <your-tidb-username>
Password: <your-tidb-password>
Database: metricguard_db
SSL:      Required
```

The MetricGuard backend constructs the SQLAlchemy URL automatically from environment variables:

```
mysql+pymysql://<DB_USER>:<DB_PASSWORD>@<DB_HOST>:<DB_PORT>/metricguard_db
```

## Step 8: Verify the Connection

### Using Python (SQLAlchemy):

```python
from sqlalchemy import create_engine, text

DB_HOST     = "gateway01.ap-southeast-1.prod.aws.tidbcloud.com"
DB_PORT     = "4000"
DB_USER     = "<your-tidb-username>"
DB_PASSWORD = "<your-tidb-password>"
DB_NAME     = "metricguard_db"

DATABASE_URL = f"mysql+pymysql://{DB_USER}:{DB_PASSWORD}@{DB_HOST}:{DB_PORT}/{DB_NAME}"

engine = create_engine(
    DATABASE_URL,
    connect_args={"ssl": {"ssl_verify_cert": False, "ssl_verify_identity": False}},
)

try:
    with engine.connect() as conn:
        conn.execute(text("SELECT 1"))
    print("✅ Successfully connected to TiDB Cloud!")
except Exception as e:
    print(f"❌ Connection failed: {e}")
```

### Using the MySQL CLI:

```bash
mysql -h gateway01.ap-southeast-1.prod.aws.tidbcloud.com \
      -P 4000 \
      -u <your-tidb-username> \
      -p \
      --ssl-mode=REQUIRED

# Once connected:
USE metricguard_db;
SHOW TABLES;
SELECT COUNT(*) FROM metrics;
```

## Step 9: Create Indexes (Recommended)

MetricGuard's SQLAlchemy models define indexes automatically. If you need to create them manually for performance tuning:

```sql
-- Index on metrics timestamp (already defined in ORM model)
CREATE INDEX idx_metrics_timestamp ON metrics (timestamp DESC);

-- Index on logs level and service_name (already defined in ORM model)
CREATE INDEX idx_logs_level        ON logs (level);
CREATE INDEX idx_logs_service_name ON logs (service_name);
CREATE INDEX idx_logs_timestamp    ON logs (timestamp DESC);

-- Index on anomalies severity
CREATE INDEX idx_anomalies_severity ON anomalies (severity);
```

## Data Retention Strategy

To keep the database lean on the free tier, run periodic cleanup against old records:

```python
from sqlalchemy.orm import Session
from sqlalchemy import delete
from datetime import datetime, timedelta
from app.database import SessionLocal
from app.models import Metric, Log

db: Session = SessionLocal()
cutoff = datetime.utcnow() - timedelta(days=30)

# Delete metrics older than 30 days
result = db.execute(delete(Metric).where(Metric.timestamp < cutoff))
db.commit()
print(f"Deleted {result.rowcount} old metric records")

# Delete logs older than 14 days
log_cutoff = datetime.utcnow() - timedelta(days=14)
result = db.execute(delete(Log).where(Log.timestamp < log_cutoff))
db.commit()
print(f"Deleted {result.rowcount} old log records")

db.close()
```

## Troubleshooting

| Problem | Solution |
|---------|----------|
| `OperationalError: (2003) Can't connect to MySQL server` | Check `DB_HOST` and `DB_PORT`; ensure network access is allowed |
| `Access denied for user` | Verify `DB_USER` and `DB_PASSWORD`; re-generate password in TiDB Cloud Console |
| `SSL connection error` | Ensure `ssl_verify_cert: False` is set in SQLAlchemy `connect_args` for Serverless |
| Connection works locally but not in Docker | Use the full TiDB Cloud hostname — do not use `localhost` or `127.0.0.1` |
| `Table doesn't exist` | Run `Base.metadata.create_all(bind=engine)` or restart the backend to trigger auto-migration |
| Slow queries | Check missing indexes; use TiDB Cloud SQL Editor to run `EXPLAIN` on slow queries |
| Storage approaching limit | Run the data retention cleanup script above |
